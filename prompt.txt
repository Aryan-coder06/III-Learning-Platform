PART C — RAG-FIRST BACKEND FOUNDATION
==================================================

Now build the backend with a ROOM-SCOPED RAG architecture.

This is the most important part.

The RAG system must work per private room.

Meaning:
- uploaded PDFs and files belong to a room
- extracted chunks belong to a room
- retrieval happens inside room context
- answers should only use indexed knowledge from that room unless cross-room access is explicitly added later

==================================================
RAG PRODUCT BEHAVIOR
==================================================

Inside a private room, users should be able to:
1. upload PDFs or notes
2. have those documents processed
3. ask questions inside that room
4. receive grounded answers from that room’s uploaded resources
5. see document/chunk source references

Examples:
- “What does our OS PDF say about deadlock?”
- “Find where normalization is explained in our DBMS notes”
- “What are the key formulas from the uploaded CN material?”

This RAG should be the first real intelligence feature.

==================================================
DYNAMIC DATA REQUIREMENT
==================================================

This is critical:

The system must be designed so RAG keeps working when new files are uploaded dynamically.

That means:
- new uploads should not require rebuilding the entire dataset
- each uploaded document should be processed independently
- only changed/new content should be chunked and indexed
- retrieval should always use the latest indexed data for the selected room

Design it as incremental RAG.

Required logic:
1. User uploads document to a room
2. Save file locally
3. Store document metadata in MongoDB
4. Extract text
5. Chunk text
6. Store chunks in MongoDB
7. Generate embeddings
8. Insert chunk embeddings into vector index
9. Mark document as indexed
10. Room RAG queries should now include the new document automatically

Also handle future update/delete flow cleanly:
- if a document is replaced, old chunks must be invalidated or removed
- if a document is deleted, its vector entries must not keep appearing in retrieval
